# Design Guide — Thunderbird Support Reports

Design system: **Bolt** (bolt.thunderbird.net) — Thunderbird's design system.
These reports are web surfaces. Apply web conventions throughout.

---

## Brief for Fable

You are a expert data analyst, storyteller, and accessible web designer with strong graphic design sensibility. Your task is to make the Thunderbird support reports significantly more beautiful, readable, and useful for data-driven decision-making — while staying true to the Bolt design system and Thunderbird's brand.

### What "better" means here
- **Readable at a glance:** A manager opening this on a phone between meetings should understand the headline story in 5 seconds. Key numbers must be prominent; supporting detail should be available but not in the way.
- **Data storytelling:** Numbers alone aren't enough. Trends, comparisons, and context should be visible — not buried in tables. Use charts, delta indicators, and callouts purposefully.
- **Information hierarchy:** Not everything is equally important. The most critical signal (blocker? spike? CSAT drop?) should dominate the visual hierarchy. Use color and weight intentionally.
- **Accessible:** WCAG AA minimum. Color is never the only signal. Text contrast meets ratio requirements against both light and dark backgrounds. Interactive elements have focus states.
- **Beautiful:** These represent Thunderbird Support externally. They should look like they were made with care — not like a script dumped data into a template.

### Hardening for lower-level models (Sonnet and below)
When redesigning, structure the HTML and CSS so that future maintenance by a less capable model is safe:
- All colors via CSS custom properties — no raw hex in component styles
- Semantic variable names that communicate intent (`--color-surface-raised`, not `--bg-card`)
- CSS variables defined in one place (`:root`) with a clear comment block explaining the palette
- Component patterns that are explicit and repetitive rather than clever — prefer clarity over DRY
- Inline style overrides should be rare; favor adding a class
- Generated HTML (from Python f-strings) should use the same CSS classes as the static shell — no special-cased inline styles per-row unless truly necessary
- Avoid deeply nested f-string lambdas where a helper function would be clearer

### Future-proofing
- Use Bolt semantic tokens (see Color section below) — when Bolt updates, only `:root` needs to change
- Design with the assumption that data volume grows: tables that have 10 rows today may have 50 next year; charts that cover 10 days will cover 180
- All report structure is generated by Python scripts. Design decisions should be captured as named CSS classes, not as one-off inline styles — so the Python can emit `class="card card--warning"` rather than `style="background:#1c1200;border-color:#f59e0b"`

## Technical constraints

- **Self-contained HTML** — each report is a single `.html` file with all CSS and JS inline. No external stylesheets, no framework, no build step. Chart.js is loaded from CDN.
- **Generated by Python** — HTML is assembled via f-strings in Python scripts. CSS lives in a `<style>` block at the top. Any redesign must be implementable as Python string output.
- **Dark mode by default** — all current reports use dark backgrounds. Maintain this; it suits the data-dense, always-on nature of these dashboards.
- **No component library** — patterns are hand-authored. Create clear, repeatable CSS class patterns that a Python script can reliably emit.
- **Font:** Inter (from Google Fonts CDN, already loaded in reports)

## Principles

- **Consistent familiarity** — simplicity, ease of use, logical structure
- **Inclusive freedom** — accessible, adaptable, customizable
- **Privacy companion** — privacy-first, transparent, respectful of user data and attention
- **Humanize user collaboration** — clear feedback, supportive tone, jargon-free
- **Craft** — clean, minimal, elegant; perceivable quality across platforms
